# Self-Discovering CI/CD Platform

Complete architecture and design for an automatic, scalable CI/CD platform that discovers and deploys services without manual pipeline configuration.

## Problem Statement

**Current Flow (Manual):**
- Create new service folder
- Create Dockerfile, Chart.yaml, service code
- Modify CI workflow to add build/deploy steps
- Update ArgoCD Application definitions
- Commit CI changes
- **Problem:** CI complexity grows linearly with each new service

**Desired Flow (Automatic):**
- Create new service folder with metadata
- Push to Git
- CI automatically discovers, builds, and deploys
- **No CI pipeline changes ever needed**

## Service Metadata: service.yaml

Every service contains self-describing metadata:

```yaml
# services/github-summary/service.yaml

name: github-summary
description: "Daily GitHub activity summarizer"

# Build configuration
build:
  enabled: true                        # Skip build if false
  dockerfile: ./Dockerfile
  context: ./                          # Build context

# Image registry configuration
image:
  registry: ghcr.io/rprakashdass      # Container registry
  name: github-summary                # Image name

# Deployment configuration
deploy:
  enabled: true                        # Skip deploy if false
  chart: ./Chart.yaml                  # Helm chart location
  helmChart: true                      # Is a Helm chart

# Versioning strategy
versioning:
  strategy: git-sha                    # Options: git-sha, git-tag, timestamp
  autoUpdate: true                     # Auto-update values.yaml

# Optional: Domain/category
domain: automation

# Optional: Dependencies (other services that must deploy first)
dependencies: []

# Optional: Notification on build/deploy
notifications:
  enabled: false
  channels: ["slack", "email"]
```

### Design Decisions for service.yaml

**Decision: Store alongside source code**
- **Why:** Service owns its build/deploy configuration. No separate metadata repo.
- **Benefit:** Changes to service logic, Dockerfile, and metadata stay synchronized in Git.

**Decision: Explicit build/deploy flags**
- **Why:** Some services may have chart but no Dockerfile (use external images). Others build but don't auto-deploy.
- **Benefit:** Flexibility without pipeline complexity.

**Decision: Versioning strategy as first-class config**
- **Why:** Different services may need different versioning (git-sha for frequent builds, git-tag for releases).
- **Benefit:** Platform scales without forcing one-size-fits-all versioning.

## CI/CD Pipeline: Stateless Discovery

The GitHub Actions pipeline is **universal and never changes**:

```yaml
# .github/workflows/build-and-deploy.yml

name: Self-Discovering Build & Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  discover:
    name: Discover Changed Services
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.discover.outputs.matrix }}
      has-changes: ${{ steps.discover.outputs.has-changes }}

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history for git diff

      - name: Detect changed services
        id: discover
        run: |
          # Get changed files from main..HEAD (or from last successful build)
          CHANGED_FILES=$(git diff --name-only origin/main HEAD 2>/dev/null || git diff --name-only HEAD~1 HEAD)

          # Extract unique service folders (first part of path in services/)
          SERVICES=$(echo "$CHANGED_FILES" | \
            grep '^services/' | \
            cut -d'/' -f2 | \
            sort -u)

          if [ -z "$SERVICES" ]; then
            echo "has-changes=false" >> $GITHUB_OUTPUT
            exit 0
          fi

          # For each service folder, check if service.yaml exists
          MATRIX_JSON=$(echo "$SERVICES" | jq -R -s -c '
            split("\n") |
            map(select(length > 0)) |
            . as $services |
            map({service: .})
          ')

          echo "matrix=$MATRIX_JSON" >> $GITHUB_OUTPUT
          echo "has-changes=true" >> $GITHUB_OUTPUT

  build:
    name: Build Services
    needs: discover
    if: needs.discover.outputs.has-changes == 'true'
    runs-on: ubuntu-latest
    strategy:
      matrix: ${{ fromJson(needs.discover.outputs.matrix) }}
      fail-fast: false  # Build all even if one fails

    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Load service metadata
        id: metadata
        run: |
          SERVICE_PATH="services/${{ matrix.service }}"

          if [ ! -f "$SERVICE_PATH/service.yaml" ]; then
            echo "ERROR: $SERVICE_PATH/service.yaml not found"
            exit 1
          fi

          # Parse YAML (requires yq or similar)
          BUILD_ENABLED=$(yq -r '.build.enabled // true' "$SERVICE_PATH/service.yaml")
          DEPLOY_ENABLED=$(yq -r '.deploy.enabled // true' "$SERVICE_PATH/service.yaml")
          REGISTRY=$(yq -r '.image.registry' "$SERVICE_PATH/service.yaml")
          IMAGE_NAME=$(yq -r '.image.name' "$SERVICE_PATH/service.yaml")
          DOCKERFILE=$(yq -r '.build.dockerfile // "./Dockerfile"' "$SERVICE_PATH/service.yaml")
          VERSION_STRATEGY=$(yq -r '.versioning.strategy // "git-sha"' "$SERVICE_PATH/service.yaml")

          echo "build-enabled=$BUILD_ENABLED" >> $GITHUB_OUTPUT
          echo "deploy-enabled=$DEPLOY_ENABLED" >> $GITHUB_OUTPUT
          echo "registry=$REGISTRY" >> $GITHUB_OUTPUT
          echo "image-name=$IMAGE_NAME" >> $GITHUB_OUTPUT
          echo "dockerfile=$DOCKERFILE" >> $GITHUB_OUTPUT
          echo "version-strategy=$VERSION_STRATEGY" >> $GITHUB_OUTPUT

      - name: Generate image tag
        id: tag
        run: |
          SERVICE=${{ matrix.service }}
          STRATEGY=${{ steps.metadata.outputs.version-strategy }}

          case "$STRATEGY" in
            git-sha)
              TAG=$(git rev-parse --short HEAD)
              ;;
            git-tag)
              # Use git tag if available, else commit SHA
              TAG=$(git describe --tags --exact-match 2>/dev/null || git rev-parse --short HEAD)
              ;;
            timestamp)
              TAG=$(date -u +%Y%m%d-%H%M%S)
              ;;
            *)
              TAG=$(git rev-parse --short HEAD)
              ;;
          esac

          echo "tag=$TAG" >> $GITHUB_OUTPUT
          echo "full-image=${{ steps.metadata.outputs.registry }}/${{ steps.metadata.outputs.image-name }}:$TAG" >> $GITHUB_OUTPUT

      - name: Set up Docker Buildx
        if: steps.metadata.outputs.build-enabled == 'true'
        uses: docker/setup-buildx-action@v3

      - name: Log in to container registry
        if: steps.metadata.outputs.build-enabled == 'true'
        uses: docker/login-action@v3
        with:
          registry: ${{ steps.metadata.outputs.registry }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push image
        if: steps.metadata.outputs.build-enabled == 'true'
        uses: docker/build-push-action@v5
        with:
          context: services/${{ matrix.service }}/${{ steps.metadata.outputs.dockerfile }}
          push: ${{ github.ref == 'refs/heads/main' }}
          tags: |
            ${{ steps.tag.outputs.full-image }}
            ${{ steps.metadata.outputs.registry }}/${{ steps.metadata.outputs.image-name }}:latest
          build-args: |
            VERSION=${{ steps.tag.outputs.tag }}

      - name: Upload image metadata
        uses: actions/upload-artifact@v4
        with:
          name: image-metadata-${{ matrix.service }}
          path: |
            services/${{ matrix.service }}/service.yaml
          retention-days: 1

  update-values:
    name: Update Helm Values
    needs: [discover, build]
    if: github.ref == 'refs/heads/main' && needs.discover.outputs.has-changes == 'true'
    runs-on: ubuntu-latest

    permissions:
      contents: write

    steps:
      - uses: actions/checkout@v4

      - name: Download image metadata
        uses: actions/download-artifact@v4
        with:
          path: artifacts

      - name: Update values.yaml with new image tags
        run: |
          # For each service that was built:
          # 1. Read the tag from CI
          # 2. Update platform/applications-chart/values.yaml
          # 3. Or update services/{service}/values.yaml depending on structure

          for service in artifacts/image-metadata-*/; do
            SERVICE_NAME=$(basename "$service" | sed 's/image-metadata-//')
            SERVICE_YAML="services/$SERVICE_NAME/service.yaml"

            if [ ! -f "$SERVICE_YAML" ]; then
              continue
            fi

            AUTO_UPDATE=$(yq -r '.versioning.autoUpdate // true' "$SERVICE_YAML")

            if [ "$AUTO_UPDATE" != "true" ]; then
              echo "Skipping $SERVICE_NAME (autoUpdate=false)"
              continue
            fi

            # The image metadata includes tag info
            # Update the corresponding values file
            # This depends on your Helm structure (see below)

            echo "Would update image tag for $SERVICE_NAME"
          done

      - name: Commit updated values
        run: |
          git config user.name "ci-bot"
          git config user.email "ci@homelab.local"

          if git diff --quiet; then
            echo "No changes to commit"
            exit 0
          fi

          git add -A
          git commit -m "ci: auto-update image tags

Generated by self-discovering CI pipeline
Services: $(git diff --cached --name-only | grep values.yaml | wc -l) updated"
          git push origin main
```

### Pipeline Design Decisions

**Decision: Universal, stateless pipeline**
- **Why:** Pipeline doesn't enumerate services. It discovers them from Git.
- **Benefit:** Adding a service requires zero pipeline changes. Scales infinitely.

**Decision: Parallel builds (matrix strategy)**
- **Why:** Each service builds independently.
- **Benefit:** 10 services build in ~time of 1 service (with enough workers).

**Decision: Git diff-based discovery**
- **Why:** Only build changed services.
- **Benefit:** Fast feedback. Don't rebuild everything on every push.

**Decision: service.yaml drives ALL decisions**
- **Why:** Single source of truth for how to build/deploy each service.
- **Benefit:** Change service behavior by editing one file. No pipeline code needed.

**Decision: Fail-fast=false in matrix**
- **Why:** If service-a build fails, don't cancel service-b build.
- **Benefit:** Get all failures in one run. Faster iteration.

## Image Tag Strategy

### Why Immutable Tags?

```yaml
# ❌ BAD: Mutable tag
image: ghcr.io/myaccount/github-summary:latest  # What version is this?

# ✅ GOOD: Immutable tags (multiple for same image)
image: ghcr.io/myaccount/github-summary:abc1234  # git-sha (exact commit)
image: ghcr.io/myaccount/github-summary:v1.2.3   # git-tag (release)
image: ghcr.io/myaccount/github-summary:20240613 # timestamp
```

**Design Decision: Default to git-sha**
- **Why:** Always know exactly which commit created the image.
- **Benefit:** Debuggable deployments. Easy rollback (point to old SHA).

### Tag Strategies

```yaml
# service.yaml - Define per service
versioning:
  strategy: git-sha        # Commit SHA (e.g., "abc1234" → "abc1234")
  autoUpdate: true
```

**git-sha (Recommended)**
```
Image: ghcr.io/myaccount/github-summary:abc1234
Matches: Exact commit in Git
Benefit: Deterministic, debuggable, easy rollback
```

**git-tag**
```
Image: ghcr.io/myaccount/github-summary:v1.2.3
Matches: Git tag (semver releases)
Use: When you release versions
```

**timestamp**
```
Image: ghcr.io/myaccount/github-summary:20240613-143245
Matches: Build time
Use: When date/time matters more than commit
```

## Values.yaml Automatic Update

### How It Works

```
CI Build:
  1. Builds github-summary
  2. Creates image tag: abc1234
  3. Updates values.yaml with new tag

Before:
  applications:
    github-summary:
      enabled: true
      image:
        tag: def5678  # Old tag

After:
  applications:
    github-summary:
      enabled: true
      image:
        tag: abc1234  # New tag (CI updated)

Git Commit:
  "ci: auto-update image tags (github-summary)"
```

### Values.yaml Location: Pattern 1 (Recommended)

**Structure:**
```
services/github-summary/
├── service.yaml
├── Chart.yaml
├── values.yaml          ← Default values
└── src/
```

**Helm Values (services/github-summary/values.yaml):**
```yaml
image:
  repository: ghcr.io/rprakashdass/github-summary
  tag: abc1234           # ← CI updates this

schedule: "0 7 * * *"
```

**ArgoCD Application (generated):**
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: home-github-summary
spec:
  source:
    repoURL: https://github.com/rprakashdass/homelab-gitops.git
    path: services/github-summary
    helm:
      values: |
        # Uses values.yaml from services/github-summary/
```

**CI Update Logic:**
```bash
# Update image tag in service's values.yaml
yq -i ".image.tag = \"$NEW_TAG\"" services/github-summary/values.yaml
git add services/github-summary/values.yaml
git commit -m "ci: auto-update image tags (github-summary)"
```

### Values.yaml Location: Pattern 2 (Alt)

**Structure:**
```
services/
  github-summary/
    ├── service.yaml
    ├── Chart.yaml
    └── src/

deployments/
  github-summary-values.yaml  ← Updated by CI
```

**CI Update Logic:**
```bash
yq -i ".image.tag = \"$NEW_TAG\"" deployments/github-summary-values.yaml
```

**Recommendation: Use Pattern 1** (values.yaml with service)
- Keeps all service config together
- Clearer dependency (who depends on whose Chart?)
- Simpler for local testing

## Git Commit Strategy

### Auto-Commit Rationale

**Design Decision: CI commits image tag updates back to Git**
- **Why:** GitOps principle — Git is source of truth.
- **Benefit:** ArgoCD sees the change automatically. Full audit trail.

### Commit Example

```
commit abc123def456
Author: ci-bot <ci@homelab.local>
Date:   Fri Jun 13 14:32:45 2024 UTC

    ci: auto-update image tags

    Built and pushed:
    - github-summary: abc1234
    - email-notifier: def5678

    Services deployed to: home namespace
```

### Avoiding Infinite Loops

**Pattern: Only commit on main branch**
```yaml
steps:
  - name: Commit updated values
    if: github.ref == 'refs/heads/main'
    run: |
      git commit -m "ci: auto-update image tags"
      git push origin main
```

**ArgoCD skips this commit automatically** (it sees the tag change, not a new app definition).

## Service Discovery Algorithm

How does the CI find all services?

### Algorithm

```
1. Get changed files from git diff
   - Compare main..HEAD (or latest build)
   - List all modified files

2. Extract service folders
   - Filter files in "services/" directory
   - Extract first path component (service name)
   - Deduplicate

3. Filter services with build enabled
   - Load service.yaml for each
   - Check build.enabled flag
   - Build only those with enabled=true

4. Create build matrix
   - Each service becomes one parallel job
   - Pass service name to build job
   - Load metadata in build step
```

### Example

**You push changes to main:**
```
Modified files:
  services/github-summary/src/main.py
  services/github-summary/Chart.yaml
  services/email-notifier/Dockerfile
  platform/base-chart/values.yaml
  README.md
```

**Discovery runs:**
```
Step 1: Extract services/
  ✓ services/github-summary/src/main.py → "github-summary"
  ✓ services/email-notifier/Dockerfile → "email-notifier"
  ✗ platform/base-chart/values.yaml (not in services/)
  ✗ README.md (not in services/)

Step 2: Deduplicate
  ["github-summary", "email-notifier"]

Step 3: Build matrix created
  - services: ["github-summary", "email-notifier"]

Result: Both services build in parallel
```

## Creating a New Service

### Example: Add email-notifier

**Step 1: Create service folder**
```bash
mkdir -p services/email-notifier/src
```

**Step 2: Create service.yaml**
```yaml
# services/email-notifier/service.yaml

name: email-notifier
description: "Send email notifications from Kubernetes events"

build:
  enabled: true
  dockerfile: ./Dockerfile

image:
  registry: ghcr.io/rprakashdass
  name: email-notifier

deploy:
  enabled: true
  chart: ./Chart.yaml

versioning:
  strategy: git-sha
  autoUpdate: true
```

**Step 3: Create Dockerfile**
```dockerfile
# services/email-notifier/Dockerfile

FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt

COPY src/ .

CMD ["python", "main.py"]
```

**Step 4: Create Chart.yaml**
```yaml
# services/email-notifier/Chart.yaml

apiVersion: v2
name: email-notifier
version: 1.0.0
appVersion: "1.0.0"

dependencies:
  - name: base-chart
    version: "1.0.0"
    repository: "file://../platform/base-chart"
```

**Step 5: Create values.yaml**
```yaml
# services/email-notifier/values.yaml

image:
  repository: ghcr.io/rprakashdass/email-notifier
  tag: abc1234

kind: deployment

replicaCount: 1

env:
  - name: SMTP_HOST
    valueFrom:
      secretKeyRef:
        name: email-notifier-secrets
        key: smtp_host
```

**Step 6: Create source code**
```python
# services/email-notifier/src/main.py

import os
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

def main():
    smtp_host = os.getenv("SMTP_HOST")
    logger.info(f"Email notifier starting (SMTP: {smtp_host})")

if __name__ == "__main__":
    main()
```

**Step 7: Commit and push**
```bash
git add services/email-notifier/
git commit -m "feat: add email-notifier service"
git push origin main
```

**That's it!**

CI will:
1. ✅ Detect change in services/email-notifier/
2. ✅ Load service.yaml
3. ✅ Build Dockerfile
4. ✅ Push to ghcr.io/rprakashdass/email-notifier:abc1234
5. ✅ Update values.yaml with new tag
6. ✅ Commit back to Git
7. ✅ ArgoCD deploys automatically

**No pipeline changes needed.**

## Scaling: From 3 to 50 Services

### Phase 1: Bootstrap (3 services)
- hello-world
- k8s-health-report
- notion-monthly

**CI Complexity:** One universal workflow

### Phase 2: Add Buildable Services (5-10 services)
- github-summary
- email-notifier
- git-sync-monitor
- backup-scheduler
- alert-aggregator

**CI Complexity:** Still one universal workflow (parallel builds handle it)

### Phase 3: Scale (20+ services)
- Multiple services per domain
- Different build strategies
- Different version strategies

**CI Complexity:** Still one universal workflow

**Example at 30 services:**
```yaml
# .github/workflows/build-and-deploy.yml
# Lines of code: ~300 (UNCHANGED since v1)

# Each push:
# - Detects changed services
# - Runs up to 30 builds in parallel
# - Takes ~5 minutes total
```

### Why Does This Scale?

```
Traditional Pipeline (hardcoded):
  if: 1 service changed
    run 1 build          (5 min)
  if: 2 services changed
    run 2 builds         (5 min each = 10 min)
  if: 30 services changed
    ??? (pipeline is spaghetti code)

Self-Discovering Pipeline:
  if: 1 service changed
    run 1 build in parallel    (5 min)
  if: 2 services changed
    run 2 builds in parallel   (5 min)
  if: 30 services changed
    run 30 builds in parallel  (5 min)

  Same pipeline code forever ✨
```

## Helm Integration

### How Services Connect to base-chart

**Each service's Chart.yaml:**
```yaml
# services/github-summary/Chart.yaml

apiVersion: v2
name: github-summary
version: 1.0.0

dependencies:
  - name: base-chart
    version: "1.0.0"
    repository: "file://../../platform/base-chart"
```

**Each service's values.yaml:**
```yaml
# services/github-summary/values.yaml

# These values are merged into base-chart
image:
  repository: ghcr.io/rprakashdass/github-summary
  tag: abc1234           # ← CI updates this automatically

kind: cronjob
schedule: "0 7 * * *"
command: ["python", "/app/main.py"]

resources:
  limits:
    memory: 256Mi
    cpu: 200m
```

**base-chart templates use these values:**
```yaml
# platform/base-chart/templates/workload.yaml

{{- if eq .Values.kind "cronjob" }}
apiVersion: batch/v1
kind: CronJob
metadata:
  name: {{ include "base-chart.fullname" . }}
spec:
  schedule: {{ .Values.schedule | quote }}
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: {{ .Chart.Name }}
            image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
            command: {{ .Values.command | toJson }}
{{- end }}
```

### Design Decision: service.yaml vs Chart.yaml

| File | Purpose | Who updates? |
|------|---------|--------------|
| **service.yaml** | Build/deploy metadata | Developer (manually) |
| **Chart.yaml** | Helm dependency definition | Developer (manually) |
| **values.yaml** | Helm values + image tags | **CI (automatically)** |

**Separation of Concerns:**
- `service.yaml`: HOW to build
- `Chart.yaml`: WHAT Helm chart to use
- `values.yaml`: CONFIGURATION (image tag, schedule, resources)

## Tag Update Strategies: Detailed

### Strategy 1: Update Service values.yaml (Recommended)

**File to update:**
```
services/github-summary/values.yaml
```

**Implementation:**
```bash
#!/bin/bash
SERVICE=$1
NEW_TAG=$2

VALUES_FILE="services/$SERVICE/values.yaml"

# Update image.tag in the service's values.yaml
yq -i ".image.tag = \"$NEW_TAG\"" "$VALUES_FILE"

git add "$VALUES_FILE"
git commit -m "ci: auto-update image tags ($SERVICE)"
git push origin main
```

**Pros:**
- Values stay with service
- Clear ownership
- Easy to see what changed

**Cons:**
- If you have 30 services, 30 commits (or batch them)

### Strategy 2: Batch Update (Multiple Services)

**Implementation:**
```bash
#!/bin/bash

# Build multiple services
declare -A SERVICES
SERVICES[github-summary]=abc1234
SERVICES[email-notifier]=def5678
SERVICES[git-monitor]=ghi9012

# Update all at once
for service in "${!SERVICES[@]}"; do
  tag="${SERVICES[$service]}"
  yq -i ".image.tag = \"$tag\"" "services/$service/values.yaml"
done

git add services/*/values.yaml
git commit -m "ci: auto-update image tags

- github-summary: abc1234
- email-notifier: def5678
- git-monitor: ghi9012"
git push origin main
```

**Pros:**
- One commit per CI run (cleaner history)
- Simpler to manage

**Cons:**
- Less granular history per service

### Strategy 3: Centralized Values File

**Alternative structure** (not recommended):
```
deployments/
├── github-summary-values.yaml
├── email-notifier-values.yaml
└── git-monitor-values.yaml
```

**Pros:**
- All deploy config in one place
- Can batch update easily

**Cons:**
- Separates config from source
- Harder to manage service ownership

**Recommendation: Use Strategy 1 (per-service values.yaml)** — keeps all service config together.

## Best Practices

### 1. Version Control First

```
✅ DO: Every build decision is in service.yaml
❌ DON'T: Hardcode registry/tag in Dockerfile
```

### 2. Use Immutable Tags

```
✅ DO:   image: ghcr.io/user/app:abc1234
❌ DON'T: image: ghcr.io/user/app:latest
```

### 3. Validate Helm Charts

```bash
# Before committing
helm lint services/*/
helm template test services/github-summary/
```

### 4. Document service.yaml

```yaml
# services/github-summary/service.yaml

# What does this service do?
name: github-summary
description: "Summarizes GitHub activity, committed daily at 7am UTC"

# Detailed build info
build:
  enabled: true
  dockerfile: ./Dockerfile   # Explicit location
  context: ./                # Build context

# Explicit image configuration
image:
  registry: ghcr.io/rprakashdass  # Required
  name: github-summary             # Required

# Version strategy must match your release process
versioning:
  strategy: git-sha    # Recommend git-sha for most services
  autoUpdate: true     # Should CI auto-update Helm values?
```

### 5. Use Resource Requests/Limits

```yaml
# services/github-summary/service.yaml

resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 500m
    memory: 512Mi
```

### 6. Keep service.yaml Simple

```
✅ DO:   30 lines, clear structure
❌ DON'T: 500 lines with obscure settings
```

## Migration Path

### From Manual to Self-Discovering

**Today (existing services):**
```
services/hello-world/
├── Chart.yaml
├── values.yaml
└── src/
```

**Add service.yaml:**
```
services/hello-world/
├── service.yaml         ← ADD THIS
├── Chart.yaml
├── values.yaml
└── src/
```

**Content of service.yaml:**
```yaml
name: hello-world
build:
  enabled: false         # Doesn't build (uses external image)
deploy:
  enabled: true
```

**For services with Dockerfile:**
```yaml
name: github-summary
build:
  enabled: true
  dockerfile: ./Dockerfile
image:
  registry: ghcr.io/rprakashdass
  name: github-summary
versioning:
  strategy: git-sha
  autoUpdate: true
```

**No other changes needed** — Git workflow stays the same.

## Summary: Design Principles

| Principle | Implementation |
|-----------|-----------------|
| **Self-describing** | service.yaml defines everything |
| **Auto-discovery** | CI finds services via Git diff |
| **Stateless** | Pipeline doesn't enumerate services |
| **Parallel** | Matrix builds run concurrently |
| **Immutable tags** | git-sha, git-tag, or timestamp |
| **GitOps** | CI commits image tags, ArgoCD deploys |
| **Scalable** | Same pipeline works for 3 or 300 services |
| **No hardcoding** | Zero service names in CI workflow |

**Your platform scales from 3 to 50 services without CI changes.** 🚀
